# 测试技能

提供软件测试的方法论、模式和最佳实践。

## 测试金字塔

```
        /  E2E  \        ← 少量，验证核心用户流程
       / 集成测试 \       ← 适量，验证模块间协作
      /  单元测试   \      ← 大量，验证单个函数/模块
```

## 单元测试原则

1. **一个测试一个断言** — 每个测试只验证一件事
2. **AAA 模式** — Arrange（准备）→ Act（执行）→ Assert（断言）
3. **测试行为而非实现** — 关注输入/输出，不关注内部细节
4. **独立运行** — 测试之间不应有依赖关系
5. **快速执行** — 单元测试应在毫秒级完成

## 测试用例设计

### 等价类划分
- 有效输入：正常范围内的值
- 边界值：最小值、最大值、临界值
- 无效输入：空值、负数、超长字符串、特殊字符

### 常见场景清单
- [ ] 正常路径（happy path）
- [ ] 空输入 / null / undefined
- [ ] 边界值（0、1、最大值）
- [ ] 错误处理路径
- [ ] 并发/竞态条件
- [ ] 超时场景

## Mock 与 Stub 策略

- **Stub**：替换外部依赖，返回固定值
- **Mock**：替换外部依赖，验证调用行为
- **Spy**：包装真实实现，记录调用信息

### 何时使用 Mock
- 外部 API 调用
- 数据库操作
- 文件系统操作
- 时间相关的逻辑

### 何时不用 Mock
- 纯函数
- 简单的数据转换
- 值对象

## TDD 流程

1. **红** — 先写一个会失败的测试
2. **绿** — 用最简代码让测试通过
3. **重构** — 在测试保护下改善代码结构
