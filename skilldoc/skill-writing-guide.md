# Skill 编写技巧指南

> 提取自 `skilldoc/index.md`，用于优化提示词时参考。

---

## 核心认知

格式不是重点，内容设计才是。真正的挑战在于如何组织 Skill 内部的逻辑。

---

## 五大设计模式

### 模式一：工具封装（Tool Wrapper）

**适用场景**：让 Agent 掌握特定库/框架的知识

**要点**：
- 不把 API 规范硬编码进系统提示，打包成 Skill 按需加载
- 用 `references/` 目录存放内部文档（如 `conventions.md`）
- 指令明确告诉 Agent **何时加载**参考文件（审查代码时 / 编写代码时）
- 适合分发内部编码规范或框架最佳实践

**示例结构**：
```
skills/api-expert/
├── SKILL.md          # 指令：何时加载、如何应用规范
└── references/
    └── conventions.md  # 具体的最佳实践列表
```

**关键指令写法**：
```
## 审查代码时
1. 加载规范参考文件
2. 对照每条规范检查用户代码
3. 对于每处违规，引用具体规则并给出修复建议

## 编写代码时
1. 加载规范参考文件
2. 严格遵循每条规范
```

### 模式二：生成器（Generator）

**适用场景**：确保每次输出的文档结构一致

**要点**：
- `assets/` 存放输出模板，`references/` 存放风格指南
- 指令充当"项目经理"角色：加载模板 → 读取风格指南 → 询问缺失变量 → 填充
- 严格的步骤编排（第一步、第二步...），确保 Agent 逐步执行
- 适合 API 文档生成、标准化提交信息、项目架构搭建

**示例结构**：
```
skills/report-generator/
├── SKILL.md
├── assets/
│   └── report-template.md   # 输出模板
└── references/
    └── style-guide.md       # 语气和格式规则
```

**关键指令写法**：
```
第一步：加载 'references/style-guide.md' 获取语气和格式规则。
第二步：加载 'assets/report-template.md' 获取所需的输出结构。
第三步：向用户询问填充模板所需的缺失信息。
第四步：按照风格指南规则填充模板。模板中的每个章节都必须出现在输出中。
第五步：以单个 Markdown 文档的形式返回完成的报告。
```

### 模式三：审查器（Reviewer）

**适用场景**：自动化代码审查或安全审计

**要点**：
- 将"检查什么"与"如何检查"分离
- 审查标准存储在 `references/review-checklist.md` 中，模块化管理
- 强制输出结构化报告：摘要 → 按严重程度分组的发现 → 评分 → 建议
- 严重程度分级：`error`（必须修复）、`warning`（应该修复）、`info`（建议考虑）
- 换一份清单就能变成完全不同的审计工具，基础设施不变

**示例结构**：
```
skills/code-reviewer/
├── SKILL.md
└── references/
    └── review-checklist.md   # 可替换的审查标准
```

**关键指令写法**：
```
第三步：将清单中的每条规则应用于代码。对于发现的每处违规：
- 记录行号（或大致位置）
- 分类严重程度：error、warning、info
- 解释为什么这是问题，而不仅仅是说明是什么问题
- 给出包含修正代码的具体修复建议

第四步：生成包含以下章节的结构化审查报告：
- **摘要**：代码的功能描述，整体质量评估
- **发现**：按严重程度分组
- **评分**：1-10 分，附简短说明
- **三大建议**：最具影响力的改进措施
```

### 模式四：反转（Inversion）

**适用场景**：防止 Agent 在需求不明确时乱猜

**要点**：
- 颠覆"用户提问、Agent 执行"的默认动态，让 Agent 先采访
- 设置不可绕过的门控指令："在所有阶段完成之前不得开始构建"
- 按阶段划分结构化问题，每次只问一个，等待回答后再进入下一阶段
- 严格的阶段顺序：问题发现 → 技术约束 → 综合输出
- 最终输出前要求用户确认并迭代

**示例结构**：
```
skills/project-planner/
├── SKILL.md
└── assets/
    └── plan-template.md   # 输出格式模板
```

**关键指令写法**：
```
你正在进行一次结构化需求访谈。在所有阶段完成之前，不得开始构建或设计。

## 第一阶段——问题发现（每次只问一个问题，等待每个回答）
按顺序提问，不得跳过任何问题。
- Q1："这个项目为用户解决什么问题？"
- Q2："主要用户是谁？他们的技术水平如何？"

## 第二阶段——技术约束（仅在第一阶段完全回答后进行）
...

## 第三阶段——综合（仅在所有问题都回答后进行）
...
```

### 模式五：流水线（Pipeline）

**适用场景**：确保复杂任务的每个步骤都被执行

**要点**：
- 指令本身就是工作流定义，带有硬性检查点
- 设置菱形门控条件："在用户确认之前，不得进入下一步"
- 只在特定步骤需要时才引入不同的参考文件，保持上下文窗口整洁
- 明确禁止跳步："不得跳过步骤，步骤失败时不得继续"

**示例结构**：
```
skills/doc-pipeline/
├── SKILL.md
├── assets/
│   └── api-doc-template.md
└── references/
    ├── docstring-style.md
    └── quality-checklist.md
```

**关键指令写法**：
```
你正在运行一个文档生成流水线。按顺序执行每个步骤。不得跳过步骤，步骤失败时不得继续。

## 第一步——解析与清点
...询问："这是你想要文档化的完整公开 API 吗？"

## 第二步——生成文档字符串
...在用户确认之前，不得进入第三步。

## 第三步——组装文档
...

## 第四步——质量检查
...报告结果。在呈现最终文档之前修复所有问题。
```

## 通用编写技巧

| 技巧 | 说明 |
|------|------|
| **善用目录结构** | `references/` 放参考文档/规范，`assets/` 放输出模板 |
| **指令要具体** | 明确告诉 Agent 何时加载哪个文件，而非一次性全部注入 |
| **步骤编号** | 用"第一步/第二步"强制顺序执行，防止 Agent 跳步 |
| **门控条件** | 用明确的禁止语句（"不得…直到…"）防止 Agent 提前行动 |
| **结构化输出** | 定义输出格式（摘要、发现、评分等），而非让 Agent 自由发挥 |
| **模式可组合** | 流水线末尾可加审查器步骤；生成器开头可用反转收集变量 |

## 决策树速查

| 你的问题 | 推荐模式 |
|---------|---------|
| 如何让 Agent 掌握特定库的知识？ | **工具封装** |
| 如何确保每次输出结构一致？ | **生成器** |
| 如何自动化代码审查/安全审计？ | **审查器** |
| 如何防止 Agent 在需求不明时乱猜？ | **反转** |
| 如何确保复杂任务每步都被执行？ | **流水线** |

---

## 模式组合示例

- **流水线 + 审查器**：流水线末尾加入审查器步骤做自我检验
- **反转 + 生成器**：生成器开头用反转模式收集必要变量，再填充模板
- **工具封装 + 审查器**：加载框架规范后，按清单审查代码

> 不要把复杂而脆弱的指令塞进单个系统提示。拆解工作流，应用正确的结构模式，构建更可靠的 Agent。
